8. Writing Utilities Module & Exception Handling:
   - Create the utils module and implement exception handling. Instead of writing a custom exception,
     use Box exceptions for cleaner and simpler code.
   - Why use a utilities (utils) module?
     In a project, some functions are used repeatedly across multiple files. 
     Instead of rewriting the same code again and again, I place these reusable functions inside
     a utilities module (utils/common.py).
   - This allows us to:
     + Write the function once
     + Import and reuse it anywhere in the project
     + Keep the code clean, organized, and maintainable
   - These commonly used functions are called utility functions because they provide 
     general-purpose support for different parts of the code.
   - Instead of writing code frequently, I define it in common.py and simply import it whenever
     needed. 
   - utils → common.py: add my utility functions (Functions use frequently) and exception
     handling logic there.

    ** First Function:
      ++ read_yaml() function: This function safely reads a YAML file and makes it easier to
         use in Python code.
      ++  Reads a YAML file from a given path. 
      ++ Converts its content into a ConfigBox (so I can use dot notation like data.key).
      ++ Uses logger to show a success message
      ++ Error handling: If the YAML file is empty → raises ValueError,
      ++ If any other error happens → it raises that error

    ** Second Function:
      ++ Each component creates its own directories. 
      ++ This function is used to create those directories, including an "artifacts" folder.
      ++ Inside this folder, I define subdirectories for different stages of the pipeline,
      ++ such as "data ingestion", where the data will be stored and processed.

    ** Third Function:
      ++ Returns the file size in KB.

  • We return the YAML file as a ConfigBox so that I can access dictionary values using
     dot notation. Instead of using keys like data["key"],ConfigBox allows us to use data.key, 
     making the code cleaner, easier to read, and more convenient to work with. 

  • We use @ensure_annotations on top of each function to enforce correct input data types.
    It ensures that the arguments passed to a function match the defined type hints. 
    If an incorrect data type is provided, it raises an error immediately.
    This helps catch bugs early and makes the code more reliable and robust.

  ### Deployment Steps
    - First, I will build a Docker image of the source code. Then, I will push this image to Amazon

    - Docker builds my app into a container, ECR stores it, ECS runs it, and EC2 is the machine that executes it.

       container so it can run the same way on any computer or server.
    ++ Docker packages my project (code + libraries + dependencies) into one container.
      + It works the same on every computer
      + No “it works on my machine” problem
      + Easy to deploy on cloud
    ++ What is AWS EC2?  EC2 = a virtual computer in the cloud
      + You can run my app on it
      + Like my own server but online
    ++ What is AWS ECR?  ECR = Docker image storage
      + You store my Docker image here
      + Like a cloud “Docker hub”
    ++ What is AWS ECS? ECS = service that runs Docker containers
    + It takes my image from ECR
    + Runs it on EC2 automatically
    + Manages my containers
  + work flows: Code → Docker Image → ECR → ECS → EC2 → Running App
  + Description: About the deployment

      2. Push your docker image to ECR

      3. Launch Your EC2 

      4. Pull Your image from ECR in EC2

      5. Lauch your docker image in EC2

  + These are the complete deployment steps, forming a CI/CD pipeline. Whenever I push the code to
    GitHub, it will be automatically built and deployed to the cloud. I will implement this CI/CD
    workflow using GitHub Actions.
  1. Build Docker Image:
     In this step, I write code in the Dockerfile to build a Docker image that packages the project for deployment.
     It tells Docker how to set up the application environment and package the entire project into a container image.
    - It builds a ready-to-run environment for my text summarization project. It:
      - sets up Python environment
      - installs dependencies
      - copies your code
      - prepares ML libraries
      - runs your FastAPI app
    - This Dockerfile packages your NLP project into a container so it can be easily deployed and run on any
      machine or cloud server.
  2. Setup CI/CD deployment:
    After creating the Dockerfile, the next step is to set up CI/CD deployment. For this, I will create
    a main.yaml file inside the .github/workflows directory. This file will define the CI/CD pipeline,
    including the logic and steps required to automatically build, test, and deploy the project whenever code
    is pushed to GitHub.
    - This is a CI/CD pipeline using GitHub Actions. It automatically checks the code (CI), builds a Docker image,
      pushes the image to AWS ECR, pulls the image on EC2 instance, stops the old container, runs the latest version,
      and deploys the FastAPI application automatically.
    - This workflow automates the deployment process, so there is no need for manual deployment. Whenever code is 
      pushed to GitHub, the application is automatically updated and deployed to the cloud, making deployment faster,
      safer, and production-ready.
    - Overall, this pipeline automatically builds, stores, and deploys my Dockerized FastAPI application to AWS EC2
       using GitHub Actions.
  - GitHub Actions CI/CD workflow: In the `main.yaml` file, there are three main stages.

    1. The first stage is "Continuous Integration (CI)": In this stage, some basic commands are executed, such as
       commands for testing. The workflow listens for jobs whenever code is pushed to the `main` branch in GitHub,
       and then the deployment pipeline starts automatically.
    2. The second stage is "Continuous Delivery (CD)": After the Continuous Integration stage is completed
       successfully, the Continuous Delivery stage starts. In this step, I log in to my AWS account using credentials
       stored securely in GitHub Secrets. Then, I build the Docker image and push it to Amazon ECR.
    3. The third stage is "Continuous Deployment": In this stage, I again log in to AWS, pull the latest Docker
       image from ECR, and run it inside the EC2 instance. Before starting the new container, the workflow checks
       whether another container is already running. If a container exists, it stops and removes the old one before
       running the updated version.
    - During the first deployment, this container check is not really necessary because no container exists yet.
      However, from the second deployment onward, this step is important to ensure the old container is stopped 
      before starting the new one.
  
  ++ The `main.yaml` file is mostly reusable for other projects as well. Usually, you only need to change the
      project or container name while keeping the rest of the workflow the same.
